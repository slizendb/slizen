# Failure modes and operator actions

This document describes the v0.2 staging contract. Redis or Valkey remains the
source of truth. Slizen's cache and hotness state are disposable, and a client
must treat a failed or timed-out write exactly as it would treat an ambiguous
write sent directly to Redis.

The stable public image is currently v0.2.2:

```text
ghcr.io/slizendb/slizen@sha256:7989b6ff17659b3f1b2f1d3feec8af6422b48f1f5486eb77247a5c82ba86b627
```

v0.2.3 is a source-tree release candidate. It has no published image digest or
image-bound release evidence yet. Unless a row says otherwise, the contract
below is the common behavior of the stable v0.2.2 image and the v0.2.3
candidate. Candidate-only behavior must not be used as evidence for a v0.2.2
trial.

The important version differences are:

| Area | Stable v0.2.2 | v0.2.3 source candidate |
| --- | --- | --- |
| Cache layout | One bounded local LRU. | Bounded probationary and protected tiers share the configured global limits. |
| Hotness tracking at capacity | Reuses the next bounded admission slot, which can evict HOT tracking state and remove that key's local cache entry. | Gives a HOT victim a second chance and drops that unseen observation instead; exposes a capacity-drop counter. |
| Write/cache ordering | Advances the cache epoch before dispatch, then invalidates the local entry after the upstream call returns, including on error. A slow write can therefore overlap an old local hit. | Serializes same-stripe mutations, invalidates both tiers before dispatch, and invalidates again or safely refreshes after the reply. |
| Exact `SET key value` | Successful writes invalidate the local entry. | A successful option-free `SET` can refresh an already eligible hot cache-policy key; other outcomes remain invalidated. |
| Extra metrics | Aggregate cache miss and oversized-key drop counters; no Slizen process/runtime or active-connection series. | Adds miss reasons, configured cache-limit gauges, capacity-drop telemetry, standard Go/process collectors, and active downstream connections. |

## Failure-mode matrix

| Failure or boundary | Client-visible behavior | Readiness and recovery | Data-safety contract | Signal and operator action |
| --- | --- | --- | --- | --- |
| Slizen process crash | Existing RESP connections close. New connections are refused until the process or Pod is back. Clients must reconnect using their normal bounded retry policy. | A dead process has no health endpoint. Kubernetes restarts it; readiness becomes true only after the proxy listener is active and an upstream `PING` succeeds. Cache and hotness start empty. | Redis/Valkey data is unchanged. Local cache state is lost. The result of a write whose response was not received is ambiguous. | Alert on Pod/container restart and application connection errors. Route the application endpoint directly to the recorded origin if recovery exceeds the agreed budget; do not blindly retry a non-idempotent write. |
| Container OOM or memory-limit kill | Same connection loss as a crash; there is no graceful drain. | Kubernetes reports `OOMKilled`; readiness is absent until restart and a successful upstream check. Repeated OOMs cause an unstable endpoint. | Configured cache byte and entry limits still protect retained cache state, but they are not a process-memory limit. A parsed request and an upstream response may be fully materialized. Redis remains authoritative; in-flight writes can be ambiguous. | Alert on `container_memory_working_set_bytes` versus the Pod limit and `kube_pod_container_status_last_terminated_reason{reason="OOMKilled"}`. Return the client endpoint to origin, preserve the termination evidence, then reduce request/value exposure or raise a justified Pod limit before retrying. |
| Upstream outage | `observe` reads and all writes return `ERR upstream request failed` after bounded timeouts. In `cache` mode, fresh local hits can complete in-process; misses fail. Stale fallback is disabled by default. | `/healthz` can still be `ok`, but `/readyz` returns 503 because it performs an upstream `PING`. A standalone Service therefore loses its ready endpoint. Readiness recovers after origin connectivity and `PING` recover. | No write is acknowledged by Slizen unless Redis/Valkey acknowledged it. An errored write may nevertheless have reached the origin and is ambiguous. Do not enable stale fallback for the first staging trial. | Alert on readiness, application errors, and `rate(slizen_upstream_errors_total[5m])`. Check origin health and network policy first. If the origin is healthy but Slizen cannot reach it, restore the direct endpoint. |
| Upstream recovery | New origin-bound operations resume after connectivity returns; clients may reconnect when the Pod becomes ready. | `/readyz` becomes successful on the next probe. Existing cache entries are not globally purged merely because connectivity recovered. | Entries remain subject to their original local TTL. Direct origin writes made while Slizen was disconnected are not known to v0.2 and can remain stale until local expiry. | Confirm readiness is stable, upstream errors stop increasing, application errors return to baseline, and a known staging key agrees through Slizen and directly at origin. Purge or remain in `observe` if direct writes may have changed cached prefixes. |
| Downstream client sends `AUTH` or requires unsupported initialization | Slizen returns an explicit unsupported-command error. Depending on the library, pool creation/reconnect can fail permanently or an optional metadata command can be ignored before value commands proceed. | Slizen readiness can remain green because it proves the proxy listener and origin, not application client initialization. | Slizen v0.2 has no downstream authentication. The application must use a separate plaintext/no-downstream-`AUTH` Slizen endpoint profile; upstream credentials belong in Slizen's Secret. Network isolation is therefore mandatory for a standalone RESP Service. | Before routing, run the exact client library's pool creation, connect, forced reconnect, and integration suite. Required downstream `AUTH`, `HELLO`, `CLIENT`, or other unsupported setup is a no-go. Treat a tolerated optional command as compatible only when the real library proves deterministic recovery after the explicit error. |
| Upstream authentication or ACL denial | Readiness fails when the connection/authentication handshake or `PING` is denied. Reads fail if `GET`, `MGET`, or their required per-key `PTTL` calls are denied. Forwarded application commands can fail independently. | `/readyz` remains 503 for handshake/`PING` failures; a command-specific denial may leave readiness green. Recovery requires correcting the upstream identity/ACL or restoring the direct endpoint. | Slizen never widens the configured Redis/Valkey identity's permissions. An ACL that is sufficient for the application's direct client may still be insufficient for Slizen's `go-redis/v9` handshake, readiness, or TTL lookup. | Before routing, test the exact Slizen identity and network path with its connection/auth handshake, `PING`, `GET`, `PTTL`, `MGET`, and every routed write/pass-through command. Alert on readiness and upstream errors. Do not grant broad access merely to make the probe green. |
| Upstream or downstream requires TLS | Slizen v0.2 cannot establish a TLS connection to Redis/Valkey and cannot accept TLS from an application. A URI scheme in `upstream.address` does not add TLS. | Direct use never becomes ready against a TLS-only origin; a TLS-only downstream client cannot complete its connection. | Slizen has no TLS configuration or certificate lifecycle in v0.2. A separately reviewed private tunnel/termination component changes the network architecture and owns its own trust and failure contract. | Treat a TLS requirement as a no-go unless the platform supplies that reviewed private component and the team tests the complete path. Do not disable required TLS merely to pass staging. |
| Origin is Redis Cluster or Sentinel-managed topology | A direct Cluster node can return `MOVED`/`ASK`; Slizen reports the upstream failure rather than discovering/following topology or splitting cross-slot operations. A Sentinel address is not used for master discovery/failover. | A `PING` to one configured address can be green while key-specific operations redirect or while the actual master changes. Slizen does not repair that topology state. | Slizen v0.2 uses one direct `redis.NewClient` target. It is not a Redis Cluster client, Sentinel client, or failover controller. Redis/Valkey remains authoritative, but the proxy path can become unavailable or incomplete. | Require one standalone-compatible origin endpoint. A separately operated proxy/service may own Cluster/Sentinel topology only after its routing and failover behavior is tested with Slizen. Otherwise treat the topology as a preflight no-go and keep the application direct. |
| Ambiguous write: timeout, connection loss, or upstream error after dispatch | Client receives `ERR upstream request failed`, a timeout, or a closed connection, but cannot infer whether the mutation was applied. | Readiness reflects current origin reachability, not the outcome of the earlier write. | Both versions advance an epoch so an old overlapping origin read cannot refill after the write boundary, and both leave the local entry invalidated when the call returns. v0.2.2 performs local deletion after the upstream call; v0.2.3 invalidates before dispatch and keeps both tiers invalidated on an error or ambiguous outcome. The origin may or may not contain the mutation. | Watch `slizen_upstream_errors_total{command=...}` and application reconciliation failures. Read authoritative state or use an application idempotency/reconciliation mechanism. Never interpret Slizen's error as proof that Redis did not apply the write. |
| Local TTL expiry or direct-origin write | After local fresh TTL expiry, the next read goes upstream. A direct-origin write can remain invisible through an existing local entry until that entry expires. | Readiness is unchanged while origin is reachable. | Local TTL is capped by the origin TTL, global `cache.max_local_ttl`, and the matching policy's `max_local_ttl`. Negative caching is not implemented. Stale-on-error can extend visibility only when explicitly enabled. | Compare known staging values at both paths and watch aggregate cache misses. The v0.2.3 candidate additionally exposes `slizen_cache_miss_reasons_total{reason="not_present"}`; that series is absent in v0.2.2. Choose a per-prefix staleness budget before promotion; keep direct-write prefixes in `observe` when that budget is unacceptable. |
| Concurrent `GET` and a supported write through Slizen | A `GET` that began before or overlaps a write may return the old value even if its reply arrives later. The overlapping origin read cannot leave an old refill after the cache epoch changes. This is not a transaction or linearizable multi-command API. | Readiness is unchanged unless the origin fails. | In v0.2.2, an existing local value can still be served while the upstream write is in flight and is deleted after the call. In v0.2.3, mutation gates and pre-dispatch invalidation remove that local-hit window; a successful exact option-free `SET key value` can refresh only an eligible hot cache-policy key. Direct writes bypass both versions' protections. | Exercise the application's real read/write race in staging and require zero mismatches under the application's own ordering contract. Inspect `slizen_invalidations_total{reason="write"}` and upstream errors. Route supported writes through Slizen or do not cache the prefix. |
| Cache reaches configured byte or entry limit | Requests continue. Cold or least-recently-used entries are evicted; later reads miss and go upstream. Origin load can rise. | Readiness is unchanged. | v0.2.2 uses one bounded local LRU. v0.2.3 divides the same configured global bounds between probationary and protected tiers. In both versions, eviction affects only disposable local state and approximate cache bytes do not include all process memory. | In both versions, watch `slizen_cache_bytes`, `slizen_cache_entries`, `slizen_cache_evictions_total`, Pod working-set memory, and origin GET rate. v0.2.3 additionally exposes `slizen_cache_max_bytes` and `slizen_cache_max_entries`; for v0.2.2, compare usage with the recorded configuration outside PromQL. Stop expansion when headroom crosses the pre-agreed limit. |
| Hotness tracker reaches `max_tracked_keys` | Value-bearing commands still reach origin. Cache eligibility can change, increasing misses and origin load. | Readiness is unchanged. | v0.2.2 deterministically reuses the next tracking slot even when it is HOT; the resulting demotion removes that key's local entry. The v0.2.3 candidate preserves a HOT victim, drops that unseen observation, and marks the audit incomplete. Both behaviors remain bounded. | On v0.2.2, watch tracking evictions/audit completeness plus promotions, demotions, misses, and origin traffic. v0.2.3 additionally exposes `slizen_hotness_capacity_observations_dropped_total`. Reduce keyspace scope or deliberately raise the bounded tracker limit; never infer zero drops from an absent v0.2.2 series. |
| Multiple sidecar Pods cache the same mutable keyspace | Each application Pod can serve its own locally fresh value. A write through one sidecar does not invalidate a sibling sidecar, so replicas can temporarily return different values. | Every sidecar can remain ready; readiness does not detect cross-Pod cache divergence. Restarting one Pod only clears that Pod's disposable state. | There is no cross-sidecar invalidation or shared cache in v0.2. A sibling or direct-origin write can remain invisible for up to the other sidecar's local TTL (or stale grace when explicitly enabled). | `observe` is safe. Before cache mode, use a single routed replica, an operationally read-only prefix, or written acceptance of TTL-bounded cross-sidecar staleness for every write path. Apply the same decision during rolling-update overlap. |
| Active clients reach `proxy.max_connections` | Existing accepted connections continue; later connection attempts are closed during admission and clients observe a connection failure/retry. | Process readiness can remain green because the listener and origin are healthy. Capacity recovers as accepted connections close. | The bound limits Slizen's admitted downstream connections; it is not a queue and does not guarantee application retry safety. No Redis data is changed by rejecting a connection before command dispatch. | Budget application pool size plus reconnect bursts below the limit. v0.2.3 exposes `slizen_active_connections`; v0.2.2 needs platform socket/client-pool telemetry. Treat sustained admission failure or retry amplification as a rollback signal. |
| Command exceeds `max_command_bytes`, `max_command_args`, or `max_mget_keys` | Slizen returns a specific RESP `ERR ... exceeds ...` response and closes that connection. Commands later in the same pipeline are not dispatched. | Process readiness is unchanged. | The command is rejected before Slizen dispatches it upstream, but redcon v1.6.2 has already parsed one complete command. These limits are admission limits, not pre-allocation memory ceilings. | Count `slizen_requests_total{result="error"}` and client reconnects. Treat any such response from the staged workload as a compatibility failure; reduce the workload scope or configure a justified bounded limit. |
| Deep pipeline or large aggregate pipeline replies | Several commands can be accepted before redcon flushes their replies. Working-set memory can spike; crossing the container limit causes an OOM crash and connection loss rather than a bounded RESP error. | Readiness may stay green during growth and disappears on OOM/restart. | Per-command byte/argument/MGET limits do not cap the aggregate response buffer. Cache bounds also do not include these transient replies. Redis remains authoritative, but in-flight writes can be ambiguous after a crash. | Before routing, replay the exact maximum client pipeline depth with representative worst-case values and require measured Pod-memory headroom. Treat OOM/restart or an unbounded pipeline as no-go; the Pod limit is containment, not graceful admission. |
| Redis key exceeds 1,024 bytes | A supported command is still forwarded and its reply is returned. The key is not hotness-tracked or locally cached. | Readiness is unchanged. | Key bytes are not rewritten. Audit coverage becomes incomplete; long keys cannot consume unbounded tracking state. | Watch `slizen_hotness_oversized_observations_dropped_total` and `telemetry_complete`. The separate capacity-drop metric exists only in the v0.2.3 candidate. Do not promote a prefix whose representative audit is incomplete without an explicit operator acceptance. |
| Value exceeds a policy's `max_item_bytes` | The upstream value is returned to the client but is not retained locally. Future reads continue to origin. | Readiness is unchanged. | An older local entry for that key is removed, so an oversized replacement cannot leave the superseded value available for later stale fallback. Upstream GET/MGET replies have no separate response-byte heap cap. | A rising origin request rate without cache admission is the expected symptom; correlate cache misses, cache bytes, and Pod memory. Keep a Pod limit, bound application value sizes, or deliberately raise the per-prefix cap within the memory budget. |
| `SIGTERM` or planned Pod replacement | Slizen stops accepting work, attempts to drain admitted handlers, closes connections after completion, and rejects newly admitted work during shutdown with `TRYAGAIN Slizen is shutting down`. At `proxy.shutdown_timeout`, remaining work is canceled/closed. | Readiness is marked false as drain begins. A replacement becomes ready only after its listener and origin check succeed. | Completed replies retain their normal semantics. Any write without a received response remains ambiguous. Cache and hotness state do not migrate to the replacement. | Keep `terminationGracePeriodSeconds` greater than `proxy.shutdown_timeout`. Monitor rollout duration, connection errors, and the `proxy drain ...` logs. Clients must reconnect; investigate any forced or deadline-reached drain before expanding. |

## Optional stale fallback changes the outage contract

`cache.allow_stale_on_upstream_error=true` is an explicit availability-for-
freshness tradeoff. `GET` may return an expired retained value only inside
`cache.stale_grace`; `MGET` falls back only when every missing cache-policy key
has an eligible stale value. `observe` prefixes and cache misses without a stale
copy still fail against an unavailable origin.

Do not enable stale fallback in the first staging promotion. If a later trial
uses it, write the accepted maximum staleness and affected prefixes into the
rollout worksheet, test expiry at the grace boundary, and make the behavior
visible to the service owner.

## What v0.2 does not have

v0.2 has no direct-origin invalidation channel. There is therefore no
invalidation-channel health state to monitor: a write made directly to
Redis/Valkey simply bypasses Slizen. Server-assisted invalidation and fail-safe
cache disablement are planned work, not a current guarantee.

See [STAGING_ROLLOUT.md](STAGING_ROLLOUT.md) for the endpoint-first rollback
procedure and [STAGING_RELEASE_GATE.md](STAGING_RELEASE_GATE.md) for the
pass/partial/fail decision.
